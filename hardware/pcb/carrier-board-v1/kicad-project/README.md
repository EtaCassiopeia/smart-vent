# KiCad project — placement clean, NOT routed, NOT fab-ready

Two boards here, one per LED variant (see
[`docs/battery-carrier-board.md` §2a](../../../../docs/battery-carrier-board.md#2a-led-indicator-optional)
for the tradeoff):

| | With LED | Without LED |
|---|---|---|
| Board | `smart_vent_v1.kicad_pcb` | `smart_vent_v1_no_led.kicad_pcb` |
| Project | `smart_vent_v1.kicad_pro` | `smart_vent_v1_no_led.kicad_pro` |
| DRC report | `drc_report.json` | `drc_report_no_led.json` |
| Build with | `python3 build_board.py` | `SMART_VENT_LED=0 python3 build_board.py` |

Each `.kicad_pcb` has a matching `.kicad_pro` — open that (or just the
`.kicad_pcb` directly, KiCad doesn't require the project file to open a
board standalone) so the net-class trace/via widths from
[§5](../../../../docs/battery-carrier-board.md#5-board-layout-rules)
persist between sessions instead of resetting to defaults: `Default`
(signals) at 0.25mm track / 0.5mm via, `Power` (VBAT, VBAT_RAW, SERVO_VCC,
GND) at 0.9mm track / 0.6mm via. The project files were hand-built from
KiCad's blank template (`pcbnew` scripting doesn't expose project creation)
— if KiCad ever rewrites them on save, that's expected and fine.

Both were generated by `build_board.py` (run against KiCad 10.0.4's bundled
`pcbnew` Python module — see the script for how to invoke it; it bypasses
the GUI-only netlist-import step, which modern KiCad no longer exposes to
scripting). It loads real footprints from KiCad's stock libraries where one
exists, places everything at resolved, collision-free coordinates (see
below), and wires nets per `../smart_vent_board.py`'s connectivity.
**No copper has been routed on either.**

## Placement: resolved (issue #46's DRC findings are fixed)

The original floorplan (`smart_vent_floorplan.svg`) got the *zoning* right
— left=signal, right=power, bottom=connectors — but the exact coordinates
didn't account for real footprint courtyard sizes, and at actual sizes
several parts collided: the THT slide switch substitute for SW1 (10.37mm
wide — much bigger than the small SMD pushbutton the floorplan was sized
around) and the THT radial cap substitute for C2 (8.59mm round, vs. the
small SMD electrolytic originally planned) were the two biggest culprits.

`nudge_apart.py` solves this: it builds the board, finds every pair of
overlapping courtyards (using real, post-rotation courtyard geometry), and
iteratively pushes them apart along whichever axis needs the smaller move,
holding U1/J2/the left analog cluster (R4/R5/R6/C1) fixed since they had no
DRC findings. It converges to the coordinates now hardcoded in
`build_board.py`'s `PLACE` dicts. One footprint needed a manual follow-up
the solver doesn't know about: **J2** (the servo header) had its last pad
landing past the board's bottom edge entirely at its original anchor
position (the footprint's "position" is pad 1, not its geometric center —
a 3-pin 2.54mm-pitch vertical header's last pad sits 5.08mm away from that
anchor) and separately collided with **U1**'s placeholder footprint, which
is a tall thin single-column header utterly unrepresentative of the real
XIAO module's shape (see #45) — J2 ended up shifted sideways in x to clear
U1's column, then up in y to clear the board edge by the required 0.5mm.

Re-running DRC after these fixes, on both variants:

| | With LED (before → after) | Without LED (before → after) |
|---|---|---|
| Shorts | 3 → **0** | 1 → **0** |
| Clearance violations | 1 → **0** | 1 → **0** |
| Courtyard overlaps | 8 → **0** | 5 → **0** |
| Board-edge clearance | (masked by the above) → **0** | (masked by the above) → **0** |

What's left in `drc_report*.json` is purely cosmetic: silkscreen reference
labels overlapping each other or clipped by the board edge/solder mask —
normal for default-placed footprint text once parts are packed this
tightly, fixed by dragging labels in the PCB editor, not a routing or
electrical concern.

## Gerbers (pipeline validation only — see the warning in each folder)

`gerbers-unrouted-with-led/` and `gerbers-unrouted-no-led/` are committed
Gerber + drill exports of these same two boards. **Do not send these to a
fab** — there's no copper routed, even though placement is now clean. They
exist to confirm the export pipeline produces the JLCPCB-compatible layer
set documented in the main doc's §6; each folder has its own README
repeating this warning and the exact `kicad-cli` commands used to generate
it, so they're trivially reproducible rather than load-bearing artifacts.

To regenerate either (or once a board is actually routed):

```bash
kicad-cli pcb export gerbers --output <dest>/ <board>.kicad_pcb
kicad-cli pcb export drill   --output <dest>/ <board>.kicad_pcb
```

Once routing is done, replace these with the actual fab-ready output and
drop "unrouted" from the directory names.
