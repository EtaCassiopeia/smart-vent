# KiCad project — placement-verified, NOT routed, NOT fab-ready

Two boards here, one per LED variant (see
[`docs/battery-carrier-board.md` §2a](../../../../docs/battery-carrier-board.md#2a-led-indicator-optional)
for the tradeoff):

| | With LED | Without LED |
|---|---|---|
| Board | `smart_vent_v1.kicad_pcb` | `smart_vent_v1_no_led.kicad_pcb` |
| Project | `smart_vent_v1.kicad_pro` | `smart_vent_v1_no_led.kicad_pro` |
| DRC report | `drc_report.json` | `drc_report_no_led.json` |
| Build with | `python3 build_board.py` | `SMART_VENT_LED=0 python3 build_board.py` |

Each `.kicad_pcb` has a matching `.kicad_pro` — open that (or just the
`.kicad_pcb` directly, KiCad doesn't require the project file to open a
board standalone) so the net-class trace/via widths from
[§5](../../../../docs/battery-carrier-board.md#5-board-layout-rules)
persist between sessions instead of resetting to defaults: `Default`
(signals) at 0.25mm track / 0.5mm via, `Power` (VBAT, VBAT_RAW, SERVO_VCC,
GND) at 0.9mm track / 0.6mm via. The project files were hand-built from
KiCad's blank template (`pcbnew` scripting doesn't expose project creation)
— if KiCad ever rewrites them on save, that's expected and fine.

Both were generated by `build_board.py` (run against KiCad 10.0.4's bundled
`pcbnew` Python module — see the script for how to invoke it; it bypasses
the GUI-only netlist-import step, which modern KiCad no longer exposes to
scripting). It loads real footprints from KiCad's stock libraries where one
exists, places everything at the `place_smart_vent.py` floorplan
coordinates, and wires nets per `../smart_vent_board.py`'s connectivity.
**No copper has been routed on either.**

Each `drc_report*.json` is the DRC run against its matching placement. Both
surface real, useful findings beyond the expected "unrouted" warnings:

## Footprints with no exact stock match (placeholders, need sourcing)

| Ref | Part | Stand-in used | What's actually needed |
|---|---|---|---|
| U1 | Seeed XIAO ESP32-C6 | generic 1×7 pin header | The official Seeed XIAO footprint (KiCad's stock libraries only ship `MCU_Seeed_ESP32C3`, not the XIAO carrier module). Source from Seeed's own KiCad library or via `easyeda2kicad` against the LCSC part. |
| J1 | "SM-2P" battery connector | generic 1×2 pin header | The hobby-RC "JST-SM" 2-pin plug is not a real JST series (despite the name) and isn't in KiCad's `Connector_JST.pretty`. Needs a footprint sourced from the connector's actual datasheet/LCSC listing, or substitute a connector that *is* in-library (e.g. a real `JST_PH` 2-pin) and adjust the BOM. |
| C2 | 470 µF / 10 V electrolytic | `CP_Radial_D8.0mm_P3.50mm` (THT) | The SMD `CP_Elec_6.3x5.4` footprint the original netlist specified isn't in this KiCad install's stock library — confirm it exists in a newer/different library, or use a THT radial cap (changes board-side clearance near J2). |

## Real placement problems found by DRC (not just silkscreen cosmetics)

At actual footprint sizes, the floorplan from `smart_vent_floorplan.svg` is
too tight in several spots on **both** variants — these DRC runs found
genuine copper-to-copper shorts, not just overlapping silkscreen text.

**With LED** (`drc_report.json`): 3 shorts, 1 clearance violation, 8
courtyard overlaps.
- **Shorts**: C2 GND pad ↔ C3 SERVO_VCC pad; D1 GND pad ↔ C5 V3V3 pad; D1
  LED_DIN pad ↔ C5 GND pad.
- **Clearance violation**: R1 (VBAT) ↔ R2 (NFET_GATE).
- **Courtyard overlaps**: R1↔R2, SW1↔J1, Q1↔Q2, R7↔D1, D1↔C2, D1↔C5,
  R3↔C2, C2↔C3.

**Without LED** (`drc_report_no_led.json`): 1 short, 1 clearance violation,
5 courtyard overlaps — confirms that omitting D1/R7/C5 removes most of the
worst cluster.
- **Shorts**: C2 GND pad ↔ C3 SERVO_VCC pad (the one short that has nothing
  to do with the LED — still needs fixing on both variants).
- **Clearance violation**: R1 (VBAT) ↔ R2 (NFET_GATE) — same as above,
  unrelated to the LED.
- **Courtyard overlaps**: R1↔R2, SW1↔J1, Q1↔Q2, R3↔C2, C2↔C3.

On the with-LED board, the cluster around D1/C2/C3/C5/R7 (bottom-right, the
LED + servo-bulk-cap area) is the worst offender — those five parts need
real breathing room. The R1/R2, SW1/J1, Q1/Q2, R3/C2/C3 overlaps are
present on both variants and are unrelated to the LED — those need fixing
regardless of which variant you build (tracked in #46). Silkscreen-only
findings (`silk_overlap`/`silk_over_copper`/`silk_edge_clearance`) are
cosmetic and not included in the counts above.

## What this project is useful for

Open it in KiCad's PCB editor to skip straight to: nudge the parts listed
above apart (the floorplan's *zones* — left=signal, right=power,
bottom=connectors — are still the right idea; only the exact coordinates
need spreading out), swap in real footprints for U1/J1/C2 once sourced,
then route per the five rules in
[`../../../../docs/battery-carrier-board.md`](../../../../docs/battery-carrier-board.md#5-board-layout-rules),
run DRC again, and plot Gerbers.

## Gerbers (pipeline validation only — see the big warning in each folder)

`gerbers-unrouted-with-led/` and `gerbers-unrouted-no-led/` are committed
Gerber + drill exports of these same two unrouted boards. **Do not send
these to a fab** — there's no copper routed, only placeholder placement
with the known shorts/overlaps above. They're checked in purely to confirm
the export pipeline produces the JLCPCB-compatible layer set documented in
the main doc's §6; each folder has its own README repeating this warning
and the exact `kicad-cli` commands used to generate it, so they're
trivially reproducible rather than load-bearing artifacts.

To regenerate either (or once a board is actually routed):

```bash
kicad-cli pcb export gerbers --output <dest>/ <board>.kicad_pcb
kicad-cli pcb export drill   --output <dest>/ <board>.kicad_pcb
```

Once #46 lands (real footprints + routing), replace these with the actual
fab-ready output and drop "unrouted" from the directory names.
